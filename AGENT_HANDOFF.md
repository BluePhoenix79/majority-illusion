This document is a complete, self-contained brief for continuing this project —
written so a **fresh AI coding agent (or a new teammate)** can pick it up cold.
It has three parts:

1. **Background** — what the study is and what has been built.
2. **Runbook** — exact setup, how to run the experiment, analyze results, and

3. **Copy-paste prompt** — a ready-to-use instruction block for a fresh agent.

## 1. Background

### Research question
In Retrieval-Augmented Generation (RAG), when the retrieved documents contain
**directly contradictory facts**, does the model follow the **majority** claim
simply because it appears more often — even when the majority is wrong? And does
the **ratio** of majority:minority documents, or a **chain-of-thought** prompt,
change that behavior? See `research_brief.md` for the full hypotheses (H1–H4).

### The dataset (`data/entities.json`, generated by `data/generate_dataset.py`)
- **50 fictional entities** (20 banking-themed, 30 general), each with a factual
  question, a **majority_value** and a conflicting **minority_value**, and a set
  of documents for each ratio.
- **6 document ratios** (majority:minority), spanning majority share 0.50 → 1.00:

  | ratio | docs | majority share |
  |-------|------|----------------|
  | `2:2` | 4    | 0.50 (control — no real majority) |
  | `3:2` | 5    | 0.60 |
  | `2:1` | 3    | 0.67 |
  | `3:1` | 4    | 0.75 |
  | `4:1` | 5    | 0.80 |
  | `4:0` | 4    | 1.00 (control — no conflict) |

- **Deterministic** (fixed seed). Regenerate with `python data/generate_dataset.py`.
  Adding a ratio is safe *only* if total docs ≤ 5 (there are 5 document styles);
  a 6-doc ratio (e.g. `5:1`) raises a `ValueError` by design. Regenerating does
  **not** perturb existing ratios (per-ratio RNG is seeded independently).

### The models (current)
The study compares two providers; the harness writes **one CSV row per (entity,
ratio, trial, model)**.

| provider slot | model | how it's reached |
|---------------|-------|------------------|
| `gemini`      | `gemini-3.5-flash` | native `google-genai` SDK, **Vertex AI paid tier** (service-account project auth) |
| `openrouter`  | `deepseek/deepseek-v4-flash` (default) **or** `anthropic/claude-haiku-4.5` | OpenRouter gateway (OpenAI-compatible), swap via `OPENROUTER_MODEL` |

- Both models "think"/reason before answering; the harness sets reasoning depth
  explicitly: Gemini `thinking_level=MEDIUM`; OpenRouter uses its unified
  `reasoning` field (`{"effort":"high"}` for DeepSeek, `{"max_tokens":2048}` for
  Claude). Output token budgets are generous (~2048) so reasoning tokens don't
  truncate the JSON answer.

### The harness (`harness/run_experiment.py`)
For each (entity × ratio × trial), it builds a prompt containing the conflicting
documents (deterministically shuffled per trial to control position bias), asks
for a JSON answer + a **confidence (1–5)**, queries each model, and appends a row
with full metadata to a CSV. Robustness: per-call errors are caught and recorded
in the `error` column (the run never aborts); rows are flushed after every call;
the Gemini path retries transient/network errors with backoff; the CSV is written
UTF-8 (Windows-safe). A `--mock` mode exercises the whole pipeline with no API
calls. On completion it prints a per-model token + estimated-cost summary.

### History worth knowing (so you don't repeat it)
- Model set changed several times. It is now **Gemini 3.5 Flash + OpenRouter
  (DeepSeek/Claude)**. Azure OpenAI / gpt-5-mini and the direct-Anthropic slot
  were **removed**. Old gpt-5-mini result CSVs were **deleted** as superseded.
- **Gemini auth was the hardest part.** The API key is Vertex **Express Mode**
  (`aiplatform.googleapis.com`, not AI Studio) — a plain AI Studio call gives
  `403 API_KEY_SERVICE_BLOCKED`. Express Mode's free quota (~5 req/min) got
  exhausted → `429 RESOURCE_EXHAUSTED`. Fix: **service-account project auth**
  (`genai.Client(vertexai=True, project=..., location="global")`, credentials
  via `GOOGLE_APPLICATION_CREDENTIALS`, **no** api_key — the SDK rejects
  project+api_key together). Location **must be `global`** — `gemini-3.5-flash`
  404s in `us-central1` for this project.
- Two pieces of "troubleshooting advice" received during the project were
  confidently wrong (a fake "Global Standard routing" story; a claim that you
  pass project+location *alongside* api_key). **Verify SDK/API claims against
  the installed SDK source or official docs before acting** — this bit us twice.
- Model/price facts move fast: this work happened after some training cutoffs, so
  don't assert model names or prices from memory — check `models.list()` / the
  provider's page. (Gemini 3.5 Flash and DeepSeek V4 Flash are both real and
  current as of this writing.)

See `UPDATES.md` for the full chronological log (append-only; newest at bottom).

## 2. Runbook

### 2.1 One-time setup
```bash
# from the repo root (majority-illusion/)
python -m venv .venv
./.venv/Scripts/python.exe -m pip install -r requirements.txt   # Windows
# (macOS/Linux: source .venv/bin/activate && pip install -r requirements.txt)
```

Create `.env` in the repo root (gitignored). Template:
```dotenv
# --- Gemini via Vertex AI, PAID tier (service-account project auth) ---
GEMINI_USE_VERTEX=1
GEMINI_VERTEX_PROJECT=<your-gcp-project-id>
GEMINI_VERTEX_LOCATION=global                 # must be global for gemini-3.5-flash
GOOGLE_APPLICATION_CREDENTIALS=<absolute-path-to-service-account-key.json>
# GEMINI_API_KEY=<only needed if you fall back to Express mode (no project)>

# --- OpenRouter slot (DeepSeek by default; Claude by swapping the model) ---
OPENROUTER_API_KEY=<your-openrouter-key>
OPENROUTER_MODEL=deepseek/deepseek-v4-flash   # or anthropic/claude-haiku-4.5
```
- The service-account JSON key file goes in the repo root; it is gitignored via
  the `project-*-*.json` / `credentials*.json` patterns in `.gitignore`. **Never
  commit it.** If you add a new key, confirm `git check-ignore <file>` matches.
- To use a machine's ADC instead of a key file, run
  `gcloud auth application-default login` and omit `GOOGLE_APPLICATION_CREDENTIALS`.

### 2.2 Sanity checks (no API calls)
```bash
./.venv/Scripts/python.exe -m py_compile harness/run_experiment.py
./.venv/Scripts/python.exe harness/run_experiment.py --mock --entities 3   # full pipeline, mock
```

### 2.3 Fire test (a few live calls) — always do this before a full run
```bash
./.venv/Scripts/python.exe harness/run_experiment.py --entities 2 --ratios 4:0 3:1 --output results/_fire.csv
# expect: "0 hard errors", both providers return answers + confidence; then delete _fire.csv
```
If Gemini returns `429 RESOURCE_EXHAUSTED`, the Vertex quota/auth is the problem
(see history above), not the code. If it returns `403 API_KEY_SERVICE_BLOCKED`,
the key is being used against the wrong endpoint.

### 2.4 Full runs
A full run = all 50 entities × 6 ratios × 2 providers = **600 calls**.
```bash
# one full run, standard prompting, explicit output name:
./.venv/Scripts/python.exe harness/run_experiment.py --strategy standard --output results/run_standard_1.csv

# chain multiple runs / both strategies (produces multiple CSVs for replication):
for strat in standard cot; do
  for i in 1 2 3; do
    ./.venv/Scripts/python.exe harness/run_experiment.py --strategy $strat --output results/run_${strat}_${i}.csv
  done
done
```
Useful flags: `--models gemini openrouter` (subset), `--entities N`, `--ratios 3:1 2:2`,
`--trials N` (repetitions per condition *within* one CSV), `--gemini-model`,
`--openrouter-model`, `--strategy {standard,cot}`, `--mock`, `--output PATH`.

**CSV naming convention:** `results/run_<strategy>_<n>.csv` (or let the harness
default to `results/run_<timestamp>_<runid>.csv`). Keep **standard and cot in
separate files** — the analysis code warns if a CSV mixes strategies, but don't
rely on that. Each run gets a unique `run_id` column.

### 2.5 Analyze
```bash
# token + cost accounting across all result CSVs (pure accounting, no API calls):
./.venv/Scripts/python.exe harness/token_report.py            # add --by-file for per-file

# figures (each reads the latest results/run_*.csv by default; pass --strategy to split):
./.venv/Scripts/python.exe visualizations/plot_majority_curve.py
./.venv/Scripts/python.exe visualizations/plot_flag_rate.py
./.venv/Scripts/python.exe visualizations/plot_confidence.py
./.venv/Scripts/python.exe visualizations/plot_outcome_breakdown.py
./.venv/Scripts/python.exe visualizations/plot_agreement.py
```
Scoring lives in `visualizations/common.py::classify()` — each response is one of
`MAJ` (took the majority value), `MIN` (minority), `COM` (cited both, committed to
one), `FLAG` (noted the conflict, refused to pick), `OTHER`, or `UNSCORED`
(error / unparseable). **Use this classifier, not a naive substring check** — a
naive check mislabels "cited both" answers as conflict-flagging.

### 2.6 Git workflow (IMPORTANT — repo conventions)
- **`git pull` before starting any work** — this is a 4-person team pushing
  asynchronously; never assume local is current. Review what came in.
- Commits: authored solely by the human running the session. **Do NOT add any
  Claude/AI/Co-Authored-By attribution** anywhere (commit messages, CONTRIBUTORS,
  credits). This is a project-specific rule.
- After a meaningful change, **append an entry to the end of `UPDATES.md`**
  (append-only; never edit or delete existing entries) in the format:
  ```
  ## [Mon DD, H:MM AM/PM] — <name>
  Committed: <what changed>
  Files: <files>
  Status: <done / next>
  ```
- **Never commit secrets:** `.env`, the `*.json` service-account key. Verify with
  `git status` (they must not appear) before every commit.
- Then `git pull` again (in case of new remote commits) and `git push`.
